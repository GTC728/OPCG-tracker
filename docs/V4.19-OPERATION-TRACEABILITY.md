# V4.19 — Operation Traceability

**Version:** 4.19.0  
**Status:** Shipped  
**Roadmap IDs:** R-02, R-03, 9E

## Problem

The app already recorded local actions in `auditLog` and match edits in `matchRevisions`, but:

1. Entries lacked structured actor attribution — group pulls from other members were invisible in the audit trail.
2. Mobile showed a flat audit list; desktop had no undo affordance tied to history.
3. Group foreground sync used ad-hoc 5s flush + pull intervals without shared throttling or pull timestamps.

Introducing a second event store would duplicate persistence and diverge from the existing match-centric model.

## Research & design principles

| Approach | Verdict |
|----------|---------|
| Separate event-sourcing DB | Rejected — matches remain the source of truth |
| PowerSync / second offline DB | Rejected — V4.16+ roadmap keeps Supabase + Dexie |
| Extend `auditLog` + `matchRevisions` | **Chosen** — minimal diff, same personal layer |
| Centralized foreground sync scheduler | **Chosen** — one module, throttle-aware |

**One-time solution:** enrich existing structures instead of parallel systems.

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│ Personal layer (IndexedDB)                                  │
│  auditLog[]     — append-only, MAX 80, actor + entityId     │
│  matchRevisions[] — before/after snapshots + actor          │
└─────────────────────────────────────────────────────────────┘
         ▲                              ▲
         │ appendAuditEntry()           │ editMatch / undo
         │                              │
┌────────┴────────┐            ┌────────┴────────┐
│ appStore        │            │ groupSync pull  │
│ local actions   │            │ remote actor    │
└─────────────────┘            └─────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ groupAutoSync.ts                                            │
│  runGroupForegroundSync(code, trigger)                      │
│    flush throttle: 3s  |  pull throttle: 10s                │
│    init / manual: force both                                │
│  resetGroupAutoSyncClocks() on collab teardown              │
└─────────────────────────────────────────────────────────────┘
         ▲
         │ useGroupCollab (12s interval poll)
```

### Structured audit (`R-03`)

`AuditEntry` extensions:

- `actor?: AuditActor` — `{ type: 'device' | 'user' | 'remote', id, label }`
- `entityId?: string` — related match/session id when known
- `meta?: Record<string, string>` — e.g. `updated_by` from group pull

`MatchRevision` gets optional `actor` on manual edits.

Helpers in `auditLog.ts`:

- `resolveLocalAuditActor(state)` — cloud user or device label
- `remoteAuditActor(updatedBy, label?)` — group pull attribution
- `formatAuditActor(actor)` — display helper

Group pull writes a `sync` entry when remote match updates are applied.

### Operation history panel (`R-02`)

`operationHistory.ts` merges `auditLog` with `matchRevisions` and live `matches` / `activeMatches`:

- `buildOperationHistory()` — desktop list items with revision link
- `canUndoAuditEntry()` — true for `match_complete` when match exists and is not active
- `summarizeRevisionDiff()` — human-readable field changes

`OperationHistoryPanel` (desktop `md:block`):

- Kind badges via i18n
- Expandable revision diff for `match_edit`
- Undo button calling existing `undoCompletedMatch(matchId)`

Mobile keeps the compact audit list in `SystemStatusPanel` (`md:hidden`).

### Auto sync strengthen (`9E`)

`groupAutoSync.ts` replaces raw flush+pull in `useGroupCollab`:

| Trigger | Behavior |
|---------|----------|
| `init` | Force flush + pull |
| `manual` | Force flush + pull (bypass in-flight guard) |
| `visibility` / `online` / `interval` | Throttled flush (3s) + pull (10s) |

`settings.lastGroupPullAt` records last successful pull; shown in `GroupSyncSection`.

## Verification

```bash
npm test -- src/lib/__tests__/v419Features.test.ts
npm run build
```

Manual checks:

1. Complete a match → Settings → System → Operation history shows entry with Undo.
2. Undo from history → match returns to active table; `match_undo` audit entry appended.
3. Edit a completed match → revision expandable in history.
4. Join a group → `lastGroupPullAt` updates after foreground pull.
