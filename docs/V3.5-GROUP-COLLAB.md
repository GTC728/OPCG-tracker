# V3.5 — Group Realtime Collab (Plan B)

> **狀態**：完成（V3.5.0）  
> **前置**：V3.4.x Record UX 穩定  
> **目標**：同一群組 **多人同時改**，不互相整包覆蓋；仍保持 **offline-first** 本機體驗。

---

## 為何需要 Plan B

V3.4 / 9D 現況：

- 群組 = `group_app_states` **一整包 JSON** upsert
- **最後上傳者贏**，無衝突合併、無即時
- 不適合「店裡 5–10 人同時記對局」

Plan B = **正規化資料 + 逐筆同步 + Realtime**，本機 IndexedDB 仍是 primary，雲端是協作層。

---

## 架構概覽

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Device A   │     │  Device B   │     │  Device C   │
│ IndexedDB   │     │ IndexedDB   │     │ IndexedDB   │
│ sync queue  │     │ sync queue  │     │ sync queue  │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                   │                   │
       └───────────────────┼───────────────────┘
                           ▼
              ┌────────────────────────┐
              │ Supabase Postgres      │
              │ + Realtime             │
              │ groups / sessions /    │
              │ matches / active_*     │
              └────────────────────────┘
```

### 核心原則

1. **Entity-level sync**：每筆 match / active_match / session 獨立 row，不再整包 `AppState` 上傳。
2. **Offline-first**：本機先寫入 queue，有網再 flush；UI 不阻塞。
3. **Idempotent writes**：以 `id` + `updated_at` / `revision` 樂觀鎖。
4. **Merge by ID**：衝突時同 id 比 revision；新 match 永遠 insert，不覆蓋整表。
5. **Group scope**：所有 row 帶 `group_key`（hash of group code），RLS 限成員。

## Supabase Schema（草案）

> 個人備份可保留 `app_state_snapshots`；群組協作走新表。

### 既有（保留）

- `group_members(group_key, user_id)`
- `app_state_snapshots` — 個人手動備份，V3.5 不廢

### 新增

```sql
-- 群組 metadata
create table group_profiles (
  group_key text primary key,
  display_name text,
  created_at timestamptz default now()
);

-- 場次（對應 Session）
create table sync_sessions (
  id uuid primary key,
  group_key text not null references group_profiles(group_key),
  name text not null,
  started_at timestamptz not null,
  ended_at timestamptz,
  revision bigint not null default 1,
  updated_at timestamptz not null default now(),
  updated_by uuid references auth.users(id),
  deleted_at timestamptz
);

-- 已完成對局
create table sync_matches (
  id uuid primary key,
  group_key text not null,
  session_id uuid not null references sync_sessions(id),
  match_number int not null,
  player1_id uuid not null,
  deck1_id uuid not null,
  player2_id uuid not null,
  deck2_id uuid not null,
  winner_player_id uuid not null,
  winner_deck_id uuid not null,
  first_player_id uuid,
  result_type text not null default 'normal',
  started_at timestamptz not null,
  finished_at timestamptz not null,
  notes text,
  source text not null default 'manual',
  revision bigint not null default 1,
  updated_at timestamptz not null default now(),
  updated_by uuid references auth.users(id),
  deleted_at timestamptz
);

-- 進行中對局
create table sync_active_matches (
  id uuid primary key,
  group_key text not null,
  session_id uuid not null references sync_sessions(id),
  match_number int not null,
  table_slot int,
  player1_id uuid,
  deck1_id uuid,
  player2_id uuid,
  deck2_id uuid,
  first_player_id uuid,
  started_at timestamptz not null,
  notes text,
  revision bigint not null default 1,
  updated_at timestamptz not null default now(),
  updated_by uuid references auth.users(id)
);

-- 玩家 / 牌組（群組共享 catalog，或 sync 引用）
create table sync_players (
  id uuid primary key,
  group_key text not null,
  name text not null,
  archived boolean default false,
  revision bigint not null default 1,
  updated_at timestamptz not null default now(),
  updated_by uuid references auth.users(id)
);

-- sync_deck_variants / sync_leaders — 可 Phase 2，V3.5 可先 sync player + match only，
-- deck 仍用本機 seed + group 內 uuid 映射
```

### RLS

- 所有 `sync_*` 表：`group_key IN (SELECT group_key FROM group_members WHERE user_id = auth.uid())`
- `UPDATE`：`revision = old.revision + 1` 或 app 送 `expected_revision`，不符則 409

### Realtime

- Subscribe：`sync_active_matches` filter `group_key=eq.xxx`
- Subscribe：`sync_matches` insert（完成對局即時出現在他人 History/Stats）
- 可選：`sync_sessions` 當前場次切換

## 客戶端 Sync 層

### 新模組 `src/lib/groupSync/`

| 模組 | 職責 |
|------|------|
| `syncQueue.ts` | 本機 outbox：`{ op, entity, payload, clientRevision }[]` 存 IndexedDB |
| `syncEngine.ts` | flush queue、pull remote、apply to Zustand |
| `conflictResolver.ts` | 409 → fetch remote → merge by id → retry or UI prompt |
| `realtimeBridge.ts` | Supabase channel → dispatch to store |
| `mappers.ts` | `AppState` ↔ `sync_*` rows（過渡期雙寫） |

### Store 改動

- 群組模式 ON：`createActiveMatch` → 本機 + enqueue `UPSERT active_match`
- `completeActiveMatch` → enqueue `INSERT match` + `DELETE active_match`
- 非群組模式：現有邏輯不變（純本機）

### 設定 UI

- Cloud 頁：**「群組協作模式」** toggle（加入群組後可開）
- 顯示：同步狀態（已同步 / 待上傳 N 筆 / 衝突）
- 衝突卡片：「保留本機 / 使用雲端 / 逐筆檢視」（Phase 2 可簡化為 auto-merge match-by-id）

## 遷移策略

### Phase 1 — 雙寫過渡（V3.5.0）

1. 加入群組 + 開協作模式
2. 首次：本機 `AppState` **bootstrap** 上传到 `sync_*`（one-shot migration）
3. 之後只走 entity sync
4. `group_app_states` 標 deprecated，只讀 fallback

### Phase 2 — 純 entity（V3.5.x）

- 移除整包 group upload/download UI
- 個人備份仍可用 snapshot（非協作場景）

### ID 對齊

- 本機 `createId()` 已是 uuid 格式 → 可直接作為 `sync_*`.id
- 確保跨裝置 id 穩定，不靠重新生成

## 衝突處理（Plan B 細節）

| 情境 | 策略 |
|------|------|
| 兩人各記一場新 match | 兩個不同 id → **無衝突**，都保留 |
| 兩人改同一 active_match | revision 高者勝；或 last-write-wins + toast |
| 一人完成、一人還在改 active | Realtime delete active → 他人 UI 關閉該桌 |
| 一人 soft-delete match | `deleted_at` 同步，Stats 排除 |
| 離線各自記 10 場 | 上線後 queue flush，10 insert 全成功 |

**不做**（V3.5 範圍外）：OT/CRDT 文字級 merge。

## 效能與規模（1000 用戶 / 同群組 ~10 人）

| 項目 | 估算 |
|------|------|
| Realtime 連線 | 每群組 ≤10，Supabase Realtime 可承受 |
| DB 寫入 | 每場 match ~1 insert + 1 delete active，峰值低 |
| 客戶端 | 訂閱增量，不全量 pull `AppState` |
| 費用 | Supabase Pro + Realtime；monitor API/connection |
